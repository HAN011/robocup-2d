# 18 天冲冠总代办与冲刺路线

## Summary
基于当前仓库现状与最新基线，这 18 天的主线定为：`规则优先 + A/B/C 分工不变 + 先降失球`。  
硬判断是：现在离“冠军级”最大的差距不是进攻花样，也不是 RL 上限，而是对强队时的防守组织、后场重启、禁区规则失误、以及缺少稳定的回归门槛。

按 [README.md](/home/ccczorange/robocup/README.md) 和 [分工.md](/home/ccczorange/robocup/分工.md) 的边界执行，18 天内所有代办按 6 条并行线推进：

1. `P0 对强队防守降失球`
2. `P0 评测与回归体系`
3. `P1 规则策略进攻补强`
4. `P1 对手情报与失球样本库`
5. `P1 提交物、环境、现场稳定性`
6. `P2 RL 仅做受控加分，不做主风险源`

## Key Changes / Workstreams
### 1. A 线：集成、验收、发布门槛
- 锁定统一日回归：每天至少 `HELIOS 3 场 + Cyrus2D 3 场`，全部保留结果文件和日志根目录。
- 建立统一日报接口，字段固定为：
  `版本ID、HELIOS 3场比分、Cyrus 3场比分、W-D-L、GF/GA、early goals(前300 cycle)、own restart内150 cycle失球、illegal restart次数、crash/timeout次数、3个代表性失球样本`
- 设定版本晋级硬门槛：
  - `0 crash`
  - `0 timeout`
  - `0 back_pass -> indirect_free_kick_r` 失球
  - `HELIOS` 不退化到 `2-1-0` 以下
  - `Cyrus2D` 总失球较当前锁定基线下降，且 early goal 不上升
- A 只做合版、验收、优先级排序，不替 B/C 长期接管其职责。

### 2. B 线：规则策略主攻项
优先顺序固定，不允许跳着做：
- `禁区规则失误归零`：先消掉 `back_pass_l / indirect_free_kick_r` 相关链路。
- `后场重启`：守门员拿球、门球、后场任意球的第一脚出球与第二点保护。
- `防线紧凑度`：球侧收缩、弱侧不过早散开、中后场纵向距离收短。
- `转换防守`：丢球后前 30-80 cycle 的回撤、补位、二防一保护。
- `门将策略`：冲击时机、封角、清球目标从“最靠前”改为“最安全且可推进”。
- `进攻补强`：传球目标评分、dribble 偏移收敛、射门判定更保守。
- `定位球攻防`：先做防守，再补 1-2 套最稳定的进攻套路。

### 3. C 线：数据与对手情报
- 建立失球标签库，标签固定只用：
  `set play / 反击 / 边路被打穿 / 中路直塞 / 二点球保护差 / 门前补位慢 / 证据不足`
- 每天从最新 Cyrus 3 场中抽 `至少 6 个` 失球样本，给出周期、前置裁判事件、保守标签、一句话说明。
- 固定输出 3 个核心指标趋势：
  `总失球`、`前300 cycle 失球`、`我方重启后150 cycle 内失球`
- 形成“强队代理画像”：
  - `Cyrus2D` 作为高压和惩罚失误代理
  - `HELIOS` 作为纪律性和控局代理
  - 若没有新增对手包，不额外扩展新代理池

### 4. 提交物与现场稳定性
- 第 `D12` 前完成一次干净环境复现。
- 第 `D15` 前完成一次完整提交包演练，包括运行说明、短赛验证卡、结果回收。
- 第 `D17-D18` 冻结高风险改动，只允许：
  - 参数微调
  - 明确 bug 修复
  - 交付与环境修复
- 准备现场 SOP：
  `端口冲突检查、conda 环境检查、rcssserver 版本检查、对手脚本路径检查、结果文件回收路径检查`

## 18-Day Schedule
### D1-D3：基线锁定期
- 锁定当前冠军冲刺基线与日报格式。
- 建立失球样本库和 3 个主指标。
- 先完成禁区规则失误、后场重启链路的问题定位。
- 产出：`第一版总代办看板 + 基线日报模板 + 失球样本库`

### D4-D7：防守止血期
- B 集中修：`back_pass/indirect free kick`、门将拿球、门球、后场任意球。
- C 每天验证“我方重启后 150 cycle 内失球”是否下降。
- A 只晋级真正降低失球且不引入退化的版本。
- 目标：从“灾难级失球”降到“可比赛级失球”。

### D8-D11：阵型与转换期
- B 集中修：防线紧凑度、回撤逻辑、补位顺序、弱侧站位。
- C 重点分析：`前300 cycle 早丢` 和 `边中路主要通道`。
- A 维护双门槛：`HELIOS 不退化 + Cyrus 继续降失球`。

### D12-D14：进攻与定位球期
- 在防守不回退的前提下，补传球选择、射门选择、保守推进。
- 只做高杠杆进攻项，不做大规模风格切换。
- 补最小可用定位球攻防方案。

### D15-D16：候选版筛选期
- 只保留 `2-3` 个候选版本。
- 对候选版本跑连续两轮完整短赛回归。
- 确认最优版本并准备冻结。

### D17-D18：赛前冻结期
- 冻结策略大改。
- 做环境复现、提交包演练、现场 SOP 演练。
- 最终只允许小参数和明确 bug 修复。

## Test Plan / Acceptance
每日验收只看这 5 件事：
- `crash/timeout` 是否归零
- `HELIOS` 是否维持竞争力
- `Cyrus2D` 总失球是否继续下降
- `前300 cycle` 是否仍稳定早丢
- `我方重启后150 cycle` 内失球是否下降

冠军冲刺版在 D16 前必须满足：
- 最近两轮回归都无异常结束
- `back_pass -> indirect_free_kick_r` 失球清零
- Cyrus 失球不再主要集中在前 300 cycle 和我方后场重启
- HELIOS 结果保持稳定，不因防守修复而明显失去进攻能力

## Interfaces / Coordination Artifacts
新增但固定的协作接口只有两个：
- `日报模板`：A 负责填，B/C 必须按同一字段回传
- `失球样本卡`：C 负责产出，A/B 每天按样本卡决定次日只修 1-2 类问题

## Assumptions
- 官方栈仍锁定 `Python + Pyrus2D`，不做底层迁移。
- RL 仍只控制 `10 号球员`，18 个 action ID 视为冻结接口。
- 18 天内默认是 `2-3 人并行`，不是全员全天候冲刺。
- 不改变 A/B/C ownership，只提高每日集成频率和门槛。
- 若没有新增强队包，`Cyrus2D + HELIOS` 就是这 18 天的主要代理对手池。
- 本周期内不做大规模 RL 重训赌博，不做 vendor 级重写，不做高风险架构改造。
